# AGENTS.md

Этот файл фиксирует общий замысел и правила для разработки тестового проекта — **монорепозитория крипто‑казино на Sepolia**. Он предназначен для Codex и других разработчиков, чтобы всегда иметь перед глазами контекст и требования.

## Общее описание проекта

Мы строим простое, но прозрачное казино, где игрок может:

1. Подключить кошелёк MetaMask к сети Sepolia (Ethereum тестнет).
2. Внести депозит тестовых ETH на контракт казино.
3. Сыграть в игру — **орёл/решка** (coin flip) — и выиграть или проиграть.
4. Вывести внутренний баланс обратно на свой кошелёк.

Казино должно иметь **математическое преимущество** (house edge) и полностью прозрачно работать на блокчейне. Каждая ставка и результат фиксируются в событиях контракта. Мы не работаем с реальными деньгами — только с Sepolia test ETH.

## Технологический стек

- **Backend/смарт‑контракты:** язык Solidity ^0.8.24, фреймворк Foundry.
- **Безопасность и библиотеки:** использовать OpenZeppelin (`Ownable`, `Pausable`, `ReentrancyGuard`), строгую логику учёта, проверки инвариантов, кастомные ошибки вместо `require` со строками.
- **Рандом:** по возможности Chainlink VRF v2.5 для получения проверяемого рандома; если это не успеваем реализовать, описать fallback (commit‑reveal или псевдослучайность на основе blockhash) и оговорить в README, что для продакшена нужно заменить.
- **Front‑end:** Next.js (App Router) с TypeScript, wagmi/viem для взаимодействия с контрактом, Tailwind CSS для стилей, pnpm для управления пакетами. Деплой планируется на Vercel.

## Смарт‑контракт CoinFlipCasino

- Контракт хранит внутренние балансы игроков, общий банк казино и данные о незавершённых ставках.
- Пользователь вносит депозит с помощью `deposit()`.
- Пользователь может вывести часть или весь баланс через `withdraw(uint256 amount)` (защита от реэнтранса).
- Метод `placeBet(Side side, uint256 amount)` создаёт заявку на игру и списывает указанную сумму с внутреннего баланса игрока.
- Рандом определяется через асинхронное обращение к Chainlink VRF. После получения случайного числа вызывается функция обратного вызова, которая сравнивает выбранную пользователем сторону с полученным результатом, начисляет выигрыш (ставка × 1.9) или оставляет депозит казино.
- Контракт эмитирует события: `Deposited`, `Withdrawn`, `BetPlaced`, `BetResolved`, `BankrollFunded`, `BankrollWithdrawn`.
- Контракт должен использовать модификаторы `nonReentrant`, `whenNotPaused` и управление доступом (owner/admin) для функций, влияющих на банк.
- Запретить размещение ставки, если баланс игрока меньше ставки или банк не может покрыть максимальную выплату (1.9 × ставка).

## Требования безопасности

1. **Порядок checks‑effects‑interactions:** сперва проверяем условия (require/castom errors), затем обновляем состояние, и только в конце вызываем внешние функции (например, отправку ETH).
2. **Reentrancy Guard:** использовать модификатор `nonReentrant` в функциях `withdraw` и других, где происходит внешнее обращение.
3. **Pausable:** контракт должен уметь остановить все критические действия (deposit, bet, withdraw) в случае проблемы.
4. **Custom Errors:** по возможности описывать ошибки как `error InsufficientBalance(uint256 requested, uint256 available);` и т.д. Это экономит газ и делает код чище.
5. **Учёт:** поддерживать инвариант: `address(this).balance >= сумма внутреннего баланса всех игроков + максимум потенциальных выплат`. Писать тесты на этот инвариант.

1. **Фаза 1:** Монорепозиторий. Настроить базовую структуру, Next.js scaffold, Foundry, README, .env.example, pnpm‑workspace.
2. **Фаза 2:** Реализация `CoinFlipCasino` по требованиям выше. Определить структуры данных, события, функцию VRF request/fulfill.

7. **Фаза 7:** Полировка UX/UI: загрузочные состояния, ошибка сети, мобильная адаптация, информирование о house edge.
8. **Фаза 8:** Финализация README, запись Loom‑видео, подготовка Notion‑страницы.
